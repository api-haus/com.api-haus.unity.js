# unity.js Package Architecture

## Package Structure

```
Packages/com.api-haus.unity.js/
├── Runtime/
│   ├── JsRuntime/Core/           — QuickJS VM, module loader, search paths
│   │   ├── JsRuntimeManager.cs   — VM singleton, script loading, CallTick/CallFunction
│   │   ├── JsScriptSearchPaths.cs — priority-ordered script source registry
│   │   ├── JsBuiltinModules.cs   — synthetic ES modules (unity.js/ecs, unity.js/math, etc.)
│   │   └── FileSystemScriptSource.cs — filesystem-backed script source
│   ├── JsECS/
│   │   ├── Components/           — JsEntityId, JsScript, JsEvent, JsTickGroup
│   │   ├── Core/                 — bridges (ECS, Query, ComponentStore, EntityRegistry)
│   │   ├── Systems/
│   │   │   ├── JsScriptingSystem.cs    — SystemBase: ticks Variable entity scripts, events
│   │   │   ├── JsSystemRunner.cs       — ISystem: system discovery, Component ticking
│   │   │   ├── Support/
│   │   │   │   ├── JsComponentInitSystem.cs  — processes JsScript(stateRef=-1)
│   │   │   │   └── JsScriptCleanupSystem.cs  — cleanup on entity destruction
│   │   │   └── Tick/             — ISystem tick systems per group
│   │   └── Authoring/            — JsScriptAuthoring, JsScriptBufferAuthoring (baker)
│   └── QJS/                      — QuickJS P/Invoke bindings
├── Editor/                       — TscCompiler, TscFileWatcher, JsTypeStubGenerator
├── Integrations/                 — CharacterController, Physics, InputSystem, etc.
├── TypeDefinitions~/             — hand-written .d.ts (globals, modules)
└── JsGameCodegen~/               — source generator for [JsBridge] component bridges
```

## Script Lifecycle

### Component Scripts

```
Editor: JsScriptAuthoring on GameObject
  ↓ baking
Baker: JsScriptBufferAuthoring writes JsScript { stateRef = -1 }
  ↓ runtime
JsComponentInitSystem: loads module, calls __componentInit, sets stateRef
  ↓ each frame
JsSystemRunner: TickComponents("update", dt) → JS __tickComponents → update(dt)
  ↓ entity destroyed
JsScriptCleanupSystem: calls onDestroy, releases VM state
```

### System Scripts

```
JsSystemRunner.DiscoverAndLoadSystems
  → scans systems/*.js in all search paths
  → loads each via LoadScriptAsModule
  → calls onUpdate(state) each frame via CallFunction
```

## Key Design Decisions

- **ISystem for hot-path systems**: JsSystemRunner and tick systems are `partial struct : ISystem` with NativeContainer state on the system entity (`JsSystemRunnerData`). Zero managed GC allocations on the hot path.
- **String interning**: `JsRuntimeManager.Intern(FixedString64Bytes)` caches managed strings — one allocation per unique script name across VM lifetime.
- **Pre-cached byte arrays**: Function names (`onUpdate`, `onTick`), tick group names (`update`, `fixedUpdate`) stored as `static readonly byte[]` to avoid per-frame `Encoding.UTF8.GetBytes`.
- **JsScript as IBufferElementData**: Changed from `ICleanupBufferElementData` so bakers can add it directly. Cleanup handled by `JsScriptCleanupSystem`.
- **No JsScriptRequest**: Baker writes `JsScript` directly with `stateRef = -1`. `JsComponentInitSystem` processes uninitialised entries. Eliminates the old two-buffer indirection.

## TypeScript Type System

Types are generated and assembled in `Library/unity.js/types/` on every domain reload:

1. `globals.d.ts` — copied from `TypeDefinitions~/` (float2/3/4, ComponentAccessor, QueryBuilder, etc.)
2. `modules.d.ts` — copied from `TypeDefinitions~/` (module declarations for unity.js/*)
3. `unity.d.ts` — auto-generated by `JsTypeStubGenerator` from `[JsBridge]` C# types

A `tsconfig.json` is auto-generated at project root including all `.ts` files from the game and package fixtures.
